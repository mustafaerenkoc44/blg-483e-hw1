# Recommendations For Production

The current solution is intentionally optimized for a single machine and a homework-sized delivery window, but the next production step should be to split the system into three separate concerns: fetch, index, and query. Fetchers should publish normalized page documents into a durable log or queue, indexers should build and update a dedicated search store, and the query tier should read from replicas tuned for low-latency search traffic. This separation removes write contention between crawling and querying, makes failure handling more explicit, and allows each subsystem to scale according to its own bottleneck instead of scaling the whole application as one unit.

For a real deployment, I would also replace the simple permissive crawl policy with stronger operational controls: robots.txt handling, per-host politeness budgets, retry and recrawl policies, observability, and abuse protection. On the storage side, SQLite should give way to a combination of durable queueing plus a search-oriented index such as OpenSearch, Elasticsearch, or a custom inverted-index service. At that point relevance can evolve beyond raw term frequency into phrase matching, anchor text, recency, language awareness, and quality signals such as link authority, while the crawler itself can distribute work safely across many workers and regions.
